10  为什么要创建数据处理 Skill

10.1 动机

使用 Claude Code 进行数据处理时，你会发现很多操作是重复性的：

  • 每次都要说明”剔除 ST 股、剔除金融业、指标要缩尾”
  • 每次都要提醒 AI 使用 Windows 路径格式
  • 每次都要告诉它输出 .do 文件、.dta 数据和 .md 小结

将这些标准化的流程打包为一个 Skill，可以：

  • 避免重复说明：Skill 中已经包含了所有常规要求
  • 提高稳定性：Skill 定义了明确的工作流程，AI 不会遗漏步骤
  • 积累经验：每次使用后可以不断完善 Skill，沉淀最佳实践

10.2 数据处理 Skill 应该做什么

一个完善的数据处理 Skill 应具备以下功能：

  1. 自动探索数据结构 — 通过读取说明文件或导入少量样本，快速了解数据的变量、类型、编码方式
  2. 全面检查数据质量问题 — 包括但不限于：保留合并报表、保留年度数据、统一变量名、字符型转数值型、剔除金融行业样本、剔除 ST 样本、剔除异常值、去重、缩尾等
  3. 关注缺失值风险 — 合并数据集后会不会大量缺失？计算某个指标时会不会因某个变量的缺失导致大量缺失？
  4. 主动询问用户 — 对于不确定的操作，向用户提问而不是自行决定
  5. 灵活适配需求 — 判断是处理单个数据集还是合并多个数据集
  6. 按需计算变量 — 根据用户需求保留和计算指定变量
  7. 输出描述性统计 — 清洗完成后对重要变量进行描述性统计

10.3 设计考量：单个 Skill vs 多个 Skill

上述功能可以放在一个综合性的 Skill 中，也可以拆分为多个专注的 Skill：

单 Skill 方案：

  • 优点：使用简单，一个 Skill 涵盖所有功能
  • 缺点：SKILL.md 可能很长，消耗较多 token

多 Skill 方案：

  • 优点：每个 Skill 聚焦特定任务（如”数据探索”、“数据清洗”、“变量计算”），更灵活
  • 缺点：需要管理多个 Skill 之间的协作
Note本书的方案

在接下来的演示中，我们采用单 Skill 方案——将所有功能整合到一个 stata-data-cleaner Skill 中。这是因为数据处理的各个环节（探索、清洗、合并、计算）紧密关联，拆分反而增加协调成本。Skill 内部通过 5 个阶段（Phase）来组织工作流，既保持了灵活性，又避免了过度拆分。

10.4 stata-data-cleaner Skill 概览

stata-data-cleaner 是我们创建的数据处理 Skill，其核心是一个 5 阶段工作流：

Phase 1: Explore  → 快速了解数据结构（只读，不修改）
Phase 2: Diagnose → 诊断数据质量问题
Phase 3: Consult  → 向用户确认不确定的决策
Phase 4: Execute  → 生成并执行 Stata 代码
Phase 5: Document → 输出 .do、.dta、.md 文件

每个阶段的输出会指导下一阶段的工作。下一章将详细介绍这个 Skill 的创建和使用过程。
